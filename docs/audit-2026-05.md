## Executive summary

The platform has a **disciplined development discipline** (ruff strict, mypy strict, pre-commit, Bandit, Trivy gate, pinned deps, pydantic everywhere) but a **lopsided risk profile**: code-quality gates are thorough, while runtime confidence (test coverage) and container/supply-chain hardening are weak. Release velocity is high (~32 tags) but tooling for visibility (logs, metrics, traces) and dependency hygiene (Dependabot, lock file, SBOM) hasn't kept pace.

**Single biggest concern:** test coverage at **13.29% line / 3.22% branch** despite 235 tests existing. The strict static gates can't catch what tests don't exercise — the team is essentially shipping by manual operator confidence.

**Single biggest quick win:** both Docker images run as root with no `USER` directive. One-line fix per Dockerfile, large security ROI.

## 1. Code maintainability

### Strengths

- Mypy `strict=true` with only **6 `# type: ignore`** across ~13k LOC of `app/` — exceptionally low escape rate.
- Ruff rule set is broad and modern (E, W, F, I, B, C4, UP, SIM, RET, PIE, S, PTH, RUF) with thoughtful per-file relaxations (tests/ ignore S105/S106/S311).
- Pre-commit covers format/lint + secret detection (`detect-private-key`) + Hadolint; mypy runs at pre-push (correct trade-off: fast commits, gated push).
- Only **3 TODO/FIXME** in `app/`, all linked to deferred-step plans (not rot):
  - [`app/main.py:115`](../app/main.py) — FIXME(step-3/-7) auth + sessions wiring
  - [`app/models/ingestion.py:138`](../app/models/ingestion.py) — FIXME(step-7) session_id NOT NULL
  - [`app/models/ingestion.py:142`](../app/models/ingestion.py) — FIXME(step-3) user_id NOT NULL
- Pydantic v2 with `pydantic.mypy` plugin — request/response contracts are type-checked end-to-end (46 `BaseModel` subclasses).

### Risks

- **`app/api/admin/sessions.py` is 2,019 LOC** — mixes session CRUD, OSM rotation, rebuild orchestration, and archival. Five functions over 130 LOC each:
  - `patch_session()` (~154 LOC)

  - `archive_session()` (~174 LOC)
  - `_reconstruct_dispatch_target()` (~137 LOC)
  - `_rotate_osm_pbf()` (~142 LOC)

  Single-file change risk + review fatigue.
- **Test coverage 13.29% line / 3.22% branch** despite 235 tests existing. With a strict-typed codebase, the *type* of bug that escapes is precisely the runtime/business-logic kind tests catch.
- **SonarCloud is wired but dormant** — `sonar-project.properties` exists, but the workflow gate is conditional on `SONARCLOUD_ENABLED` and the variable isn't set. Paid-for analysis not running.

## 2. Code performance

### Strengths

- OTP client uses `httpx.AsyncClient` ([`app/journey/otp_client.py:131`](../app/journey/otp_client.py)) — the single most latency-sensitive integration is non-blocking.
- 46 Pydantic models ⇒ validation is C-accelerated, not hand-rolled.
- `pool_pre_ping=True` on the engine ([`app/db.py:17`](../app/db.py)) — avoids stale-connection pitfall.
- Trivy + Hadolint already gating; layer count in both Dockerfiles is reasonable (~15 web, ~6 OTP).
- BuildKit cache (`cache-from`/`cache-to: type=gha`) wired in `docker.yml`.

### Risks

- **Sync SQLAlchemy 2.0.36 under async FastAPI**: 219 `def` vs 38 `async def` (82.7% sync). Every DB-touching route blocks a worker thread. Under any meaningful concurrency, this is the throughput ceiling — not OTP, not network, but the Python event loop being held by `session.execute()`.
- **Zero Python-side caching**: no `functools.lru_cache`, no Redis client, no ETag/Cache-Control middleware. Reference data (master lists, NAP catalogue, configs) is re-fetched per request. Redis is referenced in `docker-compose.yml` comments but not wired.
- **No HEALTHCHECK** in either Dockerfile → Compose / orchestrator can't distinguish "started" from "ready"; rolling deploys are blind to in-flight readiness.
- N+1 query risk: spot-checks of `app/api/admin/sessions.py` looked clean, but the full surface wasn't traced — there's no test asserting query counts.
- No load-test artefacts under `tests/` — no signal on actual latency budget.

## 3. Security

### Strengths

- All 12 prod + 14 dev deps pinned `==` (deterministic builds, no version drift on rebuild).
- pydantic-settings + `.env.example` pattern is correct; no hardcoded credentials in `app/`.
- Bandit on every PR (`-ll` = LOW+ severity); Trivy two-pass design (table-gating + non-fatal SARIF) is post-v0.1.25 sound — see [admin-guide §6.9](admin-guide.md).
- `.trivyignore` has per-CVE justifications grounded in actual usage (CLI subprocess, localhost socket only) — 6 CVEs accepted, all in upstream `docker:29-cli`.
- 127 `Depends(...)` injections + role gates (`require_platform_admin`, `require_content_manager`, `require_logged_in`) — auth surface is centralized.
- HTTP Basic + JWT coexist (Phase-1 + Phase-2); JWT extraction checks cookie then Bearer header ([`app/security.py:97-158`](../app/security.py)).

### Risks

- **Both containers run as root** — no `USER` directive in [`docker/web/Dockerfile`](../docker/web/Dockerfile) or [`docker/otp/Dockerfile`](../docker/otp/Dockerfile). Single RCE = root in container; the web container has the docker socket mounted (per worker `otp-build` flow), so container escape on RCE = host root.
- **All 16 third-party actions are tag-pinned, none SHA-pinned** — the v5/v6/v4/`@v0.36.0` tags can be force-updated by the action publisher (or compromised). For a release pipeline that pushes to GHCR, this is the primary supply-chain attack vector.
- **No lock file** (`requirements.txt` is the only pin manifest). Transitive deps not pinned ⇒ `pip install -r` today and tomorrow can resolve different transitive trees.
- **`pip-audit` is in dev deps but not invoked in CI** ([`.github/workflows/ci.yml`](../.github/workflows/ci.yml)) — Python-layer CVEs aren't gated; only OS/library layer (via Trivy on the built image) is.
- No CodeQL or Semgrep — Bandit is solid for low-hanging fruit but doesn't catch dataflow-grade issues (taint, deserialization).
- No SBOM generation (CycloneDX/Syft).
- JWT default secret in [`app/settings.py:39`](../app/settings.py) is the literal `"change-me-in-prod-use-32-bytes-random"` — no startup assertion enforces "if PROD, this must differ from default".
- No CORS middleware configured (correct *for now* given same-origin deployment, but should be commented as intentional, not forgotten).

## 4. GitHub / DevOps integration

### Strengths

- Two-workflow split is clean: lint/type/test in [`ci.yml`](../.github/workflows/ci.yml), image build/scan in [`docker.yml`](../.github/workflows/docker.yml).
- Concurrency cancellation set on `docker.yml` (avoids redundant builds on rapid pushes).
- Coverage XML + Trivy SARIF uploaded as workflow artifacts (auditable, downloadable).
- GHCR + semver tagging stack is correct.
- Per-image `vuln-type` (`os,library` for web, `os` only for OTP) reflects actual ownership of the dep tree — thoughtful.

### Risks

- **No `.github/dependabot.yml`** → 12 prod deps + 14 dev deps + GitHub actions all updated by hand. At ~30 releases over a year, dep drift is accumulating silently.
- **No CODEOWNERS, no PR template, no issue templates** — review routing and incident-report intake are ad-hoc.
- **No release automation** (release-please / semantic-release / changesets). Tagging is manual; release-note quality depends on commit-message hygiene (which is currently good — but it's not load-bearing on automation).
- **No observability stack**: stdlib `logging` only (14 imports), no `structlog`, no `prometheus_client`, no `opentelemetry`. For a platform that runs long OTP graph builds (admin-guide §6.2 documents `OOM-killed (exit 137)` as a known case), absence of metrics is a real operational gap.
- **SonarCloud job is dormant** — `if: vars.SONARCLOUD_ENABLED == 'true'` and the var isn't set.
- Branch protection not inspectable from disk — needs `gh api` confirmation that `ci`, `docker / build-and-scan (web)`, `docker / build-and-scan (otp)`, and `hadolint` are required checks on `main`.
- `docker.yml` runtime ~30 min on tagged releases (matrix × build+scan); cache may not be hitting consistently — worth monitoring `cache-from` hit-rate in build logs.

## What's already good — don't over-correct

- The strict mypy + ruff + Bandit + Trivy + Hadolint stack is mature — don't churn it.
- `==`-pinning + Trivy SARIF + table-gating-with-severity is the right two-pass design (the v0.1.25 fix earned that).
- The two-image build matrix with per-image `vuln-type` and per-image `.trivyignore` semantics is genuinely thoughtful.
- The admin-guide is a real asset — keep it as the source of operational truth; metrics and structured logs should *augment* it, not replace it.
- Pre-commit configuration (mypy at pre-push, lint/format/secret-detection at pre-commit) reflects practical experience — keep that split.
