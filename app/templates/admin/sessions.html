Configure sources

A provider is one railway / transit operator that contributes data to this session. Each provider can publish a timetable (GTFS or NeTEx) plus optional GTFS-RT live updates, minimum-connection-time table, and stations CSV. Add as many as you need (SNCF + IDFM + Trenitalia + …) — they're combined into one OTP graph at build time, with cross-operator transfers auto-generated between stops within walking distance.

Adding a new preset (or relabelling one) is now a single Python edit; the form picks it up on next deploy with no template change needed. Through v0.1.30/30.1 the rows were hand-written here, which let "rail-focused" ship to the API layer in v0.1.30 without appearing in the form — a footgun caught only when the operator saved a multi-country session and the dropdown was missing the new option. #}

Filter runs in the otp-build container via osmium-tool, before OTP parses the PBF. It cuts the data OTP has to hold in heap, not the routing accuracy — transit-focused keeps all roads + railways + public-transport amenities. rail-focused goes further: drops every drivable road and keeps only railways + walking-only paths + station amenities. ~80 % data reduction; the only scope that fits a 10-country EU merge in 24-28 GB heap on a 47 GB box. Trade-off: OTP can't compute walking from arbitrary addresses (driveable roads are gone), so the journey UI's free-text address search loses precision — fine for station-to-station rail demonstrators. Default for new sessions: transit-focused.

{# v0.1.21 — `transitModelTimeZone`. OTP 2.9 refuses to build a graph that mixes agencies with different declared timezones (SNCF says Europe/Paris, Eurostar says Europe/Brussels). The operator picks the canonical tz here; it's baked into the generated build-config.json. Default Europe/Paris keeps single-FR sessions building unchanged. #}

All API times — in and out — are interpreted in this zone (search inputs, timetables, GTFS-RT updates, displayed itineraries). Foreign-feed times get re-rendered here: e.g. with Europe/Paris, an Eurostar London 08:01 local becomes 09:01 in the response. By design, not a bug — see OTP 2.9 docs. For a multi-country session, pick the dominant operator's tz (SNCF + Eurostar + Trenitalia FR anchored to FR station search → Europe/Paris). Changing this requires a Rebuild graph; the streetGraph cache stays valid.

{# v0.1.23 — per-session OTP build heap. Until v0.1.23 the only way to bump heap was SSHing onto the VPS and editing .env; with 12-provider sessions hitting OOM during Phase 2, that was a real footgun. Same pattern as osm_scope and otp_timezone — top-level config field, validated at save, read by the worker each tick (no worker restart needed because the value is read fresh from session.config every build, not cached). #} {# v0.1.32 — default bumped from 12g to 24g. The 12g default was right when VIATOR was France-only / single-provider, but with multi-NAP and EU-scale sessions the standard case is now ~3-8 providers and 24g handles that. The form's per-session save still lets operators dial down to 12g for IDF-only or up to 36g for 10+ providers / multi-country merges. #}

⚙ Heap ≥48 GB also requires OTP_BUILD_MEM_LIMIT in .env set to heap + ~4-12 GB native overhead (e.g. heap=64g → MEM_LIMIT=76g). cgroup cap below the heap value triggers OOM-kill before -Xmx is hit. See .env.example for the recommended pairings.

⚠ Rail-focused multi-country sessions typically need ≥24 GB heap. With the current ${selected} selection, the build is at risk of OOM during the "Intersecting unconnected areas" phase. Consider bumping heap before triggering Build.

JVM heap for the build container. Rule of thumb: ~2 GB per provider on a France-wide PBF, plus 6 GB for the street graph + ~10-15% for JVM overhead (metaspace + native). Don't request more than VPS RAM minus 8 GB headroom for kernel + other containers — Linux without swap will OOM-kill instantly on overshoot. Default 12g works for SNCF-only; bump to 24-36g for multi-NAP sessions. If the build OOM-kills at "Linking transit stops" Phase 2, that's the signal to bump.

{# v0.1.24 — per-session OTP API processing timeout. Caps how long a single journey-search request can run. Default 30s; bump to 60s/120s for multi-NAP graphs that timed out at the pre-v0.1.24 hardcoded 10s (Paris GdL → Marseille on a 13-provider France-wide graph hit this every time). #}

Cap on how long OTP can spend resolving a single search before returning "0 trips in Xms (timeout)". The pre-v0.1.24 hardcoded 10 s was too tight for France-wide multi-NAP graphs (43k stops, 13 providers, 1.1M walk transfers): Paris→Marseille ran out of budget exploring alternatives even though the direct TGV exists. Bump to 60 s if you see timeout responses on routes you expect to work. Effective on next graph rebuild (writes to router-config.json); the running otp-{sid} container picks it up at next Promote.

Build & Promote

Rebuild queues an OTP graph build. The worker picks up pending jobs every 15 s after the debounce window. Build duration depends on the bundle: ~10 min for IDF, ~30–60 min for France-wide. On success the session advances to graph_built.
Promote moves graph_built → serving and asks the worker to docker compose up + nginx -s reload so the new otp-{{ s.id }} container becomes routable at /otp/{{ s.id }}/.
